## 1. 概述

本系统采用 **移动端优先 + 云端同步** 的架构设计。考古队员在野外无网络环境下，通过移动端应用（使用SQLite本地数据库）进行数据采集；当网络恢复后，自动或手动与服务端（MySQL数据库）进行双向数据同步，确保数据一致性。

### 1.1 设计原则

- **离线优先**：移动端在无网络时可完整使用所有功能
- **最终一致性**：保证数据在网络恢复后最终达成一致
- **服务端权威**：冲突时以服务端数据为准，但保护本地未提交修改
- **可追踪性**：所有同步操作均有完整日志记录

## 2. 核心概念与术语

| 术语 | 说明 |
|------|------|
| **UP（上传）** | 数据流向：移动端 → 服务端 |
| **DOWN（下载）** | 数据流向：服务端 → 移动端 |
| **双向同步** | 一次完整同步包含 UP + DOWN 两个阶段 |
| **sync_status** | 业务表中的同步状态字段，见第3章 |
| **sync_timestamp** | 记录最后一次被服务端同步确认的Unix时间戳（毫秒） |
| **last_sync_ts** | 移动端本地存储的上次增量下载时间戳（毫秒） |
| **batch_id** | 批量同步的批次号，格式：`BATCH_{deviceId}_{timestamp}` |

## 3. 同步状态流转机制

### 3.1 sync_status 枚举值定义

每张业务表（user, excavation_site, trench, artifact, artifact_photo, stratigraphy, stratigraphy_layer, stratigraphy_photo）都包含以下两个同步字段：

| 状态值 | 常量名 | 说明 |
|--------|--------|------|
| **0** | `PENDING` | 未同步：新增记录 / 本地已修改待同步 |
| **1** | `SYNCED` | 已同步：与服务端一致 |
| **3** | `FAILED` | 同步失败：重试超过最大次数，需人工介入 |

> 注：原MySQL schema中的状态2（更新待同步）已合并到状态0，简化为只需判断0或非0。

### 3.2 状态流转图（文字版）

```
          新增记录 / 本地修改
                │
                ▼
         ┌─────────────┐
         │ sync_status=0 │◄──────────┐
         └──────┬──────┘           │
                │ 触发同步          │
                ▼                   │
     ┌────────────────────┐        │
     │  同步中（内存标记）  │        │
     └───────┬────────────┘        │
             │                     │
      ┌──────┴───────┐             │
      ▼              ▼             │
 成功 │         失败 │             │
      ▼              ▼             │
┌──────────┐   ┌──────────┐       │
│ 状态 = 1 │   │ 重试 <5  │───┐   │
└──────────┘   └────┬─────┘   │   │
                   │ 否      │是 │
                   ▼         ▼   │
             ┌──────────┐ 重试 │   │
             │ 状态 = 3 │────┘   │
             └──────────┘        │
                                 │
          用户修改已同步记录 ────┘
```

### 3.3 状态流转规则

1. **新增记录**：INSERT时 `sync_status = 0`，`sync_timestamp = NULL`
2. **修改记录**：UPDATE时 `sync_status = 0`（无论之前是什么状态），`updated_at` 更新
3. **上传成功**：服务端确认后，移动端将 `sync_status = 1`，`sync_timestamp = 当前时间戳`
4. **下载记录**：插入/更新本地记录时，强制设置 `sync_status = 1`，`sync_timestamp = 服务端返回值`
5. **同步失败**：超过最大重试次数（5次）后，设置 `sync_status = 3`

## 4. 批量上传流程（UP）

### 4.1 触发条件

- 用户手动点击「同步」按钮
- 应用进入前台且网络可用时自动触发
- 定时任务（每30分钟检查一次）

### 4.2 详细流程

```
移动端                                            服务端
  │                                                │
  │ 1. 遍历所有表 WHERE sync_status IN (0, 3)       │
  │    按表分组，每组最多100条                        │
  │                                                │
  │ 2. 为每条记录生成 sync_record（状态=待同步）      │
  │                                                │
  │ 3. 生成 batch_id = BATCH_{deviceId}_{ts}        │
  │                                                │
  │ 4. POST /api/sync/upload/batch  ──────────────►│
  │    Body: {                                       │
  │      device_id, user_id, batch_id,                │
  │      tables: {                                    │
  │        user: [...],                               │
  │        excavation_site: [...],                    │
  │        ...                                        │
  │      }                                            │
  │    }                                              │
  │                                                │
  │ 5. 更新 sync_record 状态=同步中                   │
  │                                                │
  │                                                │ 6. 事务开启
  │                                                │    逐表 UPSERT
  │                                                │    写入 operation_log
  │                                                │    更新服务端 sync_timestamp
  │                                                │ 7. 事务提交
  │                                                │
  │ 8. 响应 200 OK  ◄──────────────────────────────│
  │    Body: {                                       │
  │      success: true,                               │
  │      batch_id,                                    │
  │      results: {                                   │
  │        user:    {success: N, failed: []},        │
  │        artifact:{success: N, failed: []},         │
  │        ...                                        │
  │      },                                           │
  │      server_timestamp: 1718000000000              │
  │    }                                              │
  │                                                │
  │ 9. 事务开启                                       │
  │    - 成功的记录: sync_status=1,                   │
  │      sync_timestamp=server_timestamp             │
  │    - 失败的记录: 进入重试逻辑                      │
  │    - 更新 sync_record 状态=成功/失败              │
  │ 10. 事务提交                                      │
```

### 4.3 上传顺序（表依赖顺序）

为了保证外键约束不被破坏，上传必须按以下顺序进行：

```
1. user
2. excavation_site
3. trench
4. artifact
5. artifact_photo
6. stratigraphy
7. stratigraphy_layer
8. stratigraphy_photo
```

## 5. 全量下载流程（DOWN）

### 5.1 触发条件

- 用户首次安装并登录应用
- 用户点击「恢复数据」按钮
- 本地数据库检测到损坏（完整性校验失败）
- 管理员触发的远程重置

### 5.2 详细流程

```
移动端                                            服务端
  │                                                │
  │ 1. GET /api/sync/download/all                   │
  │    ?device_id=xxx&user_id=1  ─────────────────►│
  │                                                │
  │                                                │ 2. 查询所有业务表
  │                                                │    (is_deleted=0)
  │                                                │
  │ 3. 响应（分块/压缩）◄───────────────────────────│
  │    Body: {                                       │
  │      server_timestamp: 1718000000000,            │
  │      tables: {                                    │
  │        user: [...],                               │
  │        excavation_site: [...],                    │
  │        trench: [...],                             │
  │        artifact: [...],                           │
  │        artifact_photo: [...],                     │
  │        stratigraphy: [...],                       │
  │        stratigraphy_layer: [...],                 │
  │        stratigraphy_photo: [...]                  │
  │      }                                            │
  │    }                                              │
  │                                                │
  │ 4. 【关键】检查本地是否有 sync_status=0 的记录    │
  │    ├── 有 → 弹窗提示用户：                        │
  │    │     "存在未同步的本地修改，继续将覆盖！"      │
  │    │     ├── 用户确认 → 继续                      │
  │    │     └── 用户取消 → 终止流程                  │
  │    └── 无 → 直接继续                              │
  │                                                │
  │ 5. 事务开启                                       │
  │    - DELETE 所有业务表（TRUNCATE 或 DELETE）       │
  │    - 按依赖顺序 INSERT 所有下载记录               │
  │    - 所有记录 sync_status=1                       │
  │    - 所有记录 sync_timestamp=server_timestamp     │
  │    - 写入 sync_record 批量记录                    │
  │ 6. 事务提交                                       │
  │                                                │
  │ 7. 更新 last_sync_ts = server_timestamp           │
  │ 8. 提示用户「全量同步完成」                        │
```

### 5.3 下载顺序（与上传一致）

```
1. user → 2. excavation_site → 3. trench
   → 4. artifact → 5. artifact_photo
   → 6. stratigraphy → 7. stratigraphy_layer
   → 8. stratigraphy_photo
```

## 6. 增量下载流程（DOWN）

### 6.1 触发条件

- 每次批量上传完成后 **自动执行**
- 用户手动点击「同步」按钮（先UP后DOWN）
- 定时增量同步任务

### 6.2 详细流程

```
移动端                                            服务端
  │                                                │
  │ 1. 读取本地 last_sync_ts                        │
  │    (首次为 0 或 NULL)                            │
  │                                                │
  │ 2. GET /api/sync/download/incremental  ────────►│
  │    ?last_sync_ts=1717900000000                   │
  │    &device_id=xxx                                │
  │    &user_id=1                                    │
  │                                                │
  │                                                │ 3. 逐表查询
  │                                                │    WHERE sync_timestamp > last_sync_ts
  │                                                │    包括 is_deleted=1 的记录
  │                                                │
  │ 4. 响应  ◄──────────────────────────────────────│
  │    Body: {                                       │
  │      server_timestamp: 1718000000000,            │
  │      tables: {                                    │
  │        trench: { upsert: [...], delete: [id,..]},│
  │        artifact: { upsert: [...], delete: [...]},│
  │        ...                                        │
  │      }                                            │
  │    }                                              │
  │                                                │
  │ 5. 逐表处理（事务开启）：                          │
  │    对每张表：                                      │
  │    ┌──────────────────────────────────┐           │
  │    │ A. 处理 delete 列表              │           │
  │    │    对每个 id:                     │           │
  │    │    - 若本地 sync_status=0        │           │
  │    │      → 冲突，转入第7章策略        │           │
  │    │    - 否则 → 本地 DELETE 记录     │           │
  │    │                                  │           │
  │    │ B. 处理 upsert 列表              │           │
  │    │    对每条记录：                    │           │
  │    │    - 若本地存在且 sync_status=0  │           │
  │    │      → 冲突，转入第7章策略        │           │
  │    │    - 否则 → REPLACE INTO         │           │
  │    │      sync_status=1               │           │
  │    │      sync_timestamp=服务端值      │           │
  │    └──────────────────────────────────┘           │
  │ 6. 事务提交                                       │
  │                                                │
  │ 7. 更新 last_sync_ts = server_timestamp           │
  │ 8. 如有冲突，弹出冲突列表让用户处理                 │
```

### 6.3 增量下载的核心要点

- **软删除同步**：服务端返回的 `delete` 列表包含被标记为 `is_deleted=1` 的记录ID
- **时间戳判断**：使用 `sync_timestamp` 而非 `updated_at`，因为 `sync_timestamp` 代表服务端确认时间
- **幂等性**：重复调用增量下载接口不会导致数据错误

## 7. 冲突解决策略

### 7.1 冲突场景定义

冲突发生在 **下载阶段（DOWN）**，当满足以下条件时：

```
本地记录存在  AND  本地.sync_status = 0  AND  服务端记录有变更
```

即：**本地有未上传的修改，同时服务端这条记录也被其他人修改过**。

### 7.2 冲突检测矩阵

| 本地状态 \ 服务端操作 | 更新(UPSERT) | 删除(DELETE) |
|----------------------|-------------|-------------|
| sync_status=1（已同步） | 直接覆盖，无冲突 | 直接删除，无冲突 |
| sync_status=0（有本地修改） | **冲突A**：双端更新 | **冲突B**：本地改/服务端删 |
| sync_status=3（失败） | 同 0 处理 | 同 0 处理 |

### 7.3 冲突解决流程

```
检测到冲突
    │
    ▼
┌─────────────────────────────────┐
│ 收集所有冲突记录到冲突列表         │
│ 暂停当前下载事务（不提交、不回滚）  │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 弹出冲突处理对话框                │
│ 列表显示每条冲突：                │
│   - 表名 + 记录标识（名称/编号）   │
│   - 冲突类型（更新冲突/删除冲突）   │
│   - 本地修改时间 vs 服务端修改时间 │
└──────────────┬──────────────────┘
               │
    ┌──────────┴──────────┐
    ▼                     ▼
【用户选择】          【批量操作】
  逐条选择            全部保留本地
  ┌────┴────┐        全部采用服务端
  ▼         ▼
保留本地  采用服务端
  │         │
  ▼         ▼
强制设置    覆盖本地
sync_status=0  sync_status=1
(下次上传会
 覆盖服务端)
```

### 7.4 解决策略详解

#### 策略1：保留本地修改（默认推荐用户选择）
- 跳过该条记录的下载覆盖
- 保持本地数据不变
- `sync_status` 保持为 0
- **结果**：下次 UP 时，本地修改会上传并覆盖服务端

#### 策略2：采用服务端数据
- 用服务端数据覆盖本地
- 设置 `sync_status = 1`
- 设置 `sync_timestamp = 服务端时间戳`
- **结果**：本地修改丢失

### 7.5 自动降级策略（无用户交互场景）

在后台自动同步时，若无用户在场：

1. **默认策略**：**保留本地修改**（防止数据丢失）
2. 记录冲突到 sync_record，`error_message = "CONFLICT: need user resolution"`
3. 在应用内显示红点提示，下次用户打开时弹出冲突处理

## 8. 照片文件处理

### 8.1 设计原则

- **照片文件不存入数据库**，DB中只存储路径
- **文件传输与数据同步分离**，避免大包超时
- **先传文件，后传记录**，保证引用完整性

### 8.2 照片上传流程

```
移动端                                    文件服务/服务端
  │                                            │
  │ 1. 遍历 artifact_photo / stratigraphy_photo│
  │    WHERE sync_status=0                      │
  │    且 photo_path 为本地路径                  │
  │                                            │
  │ 2. 对每张照片：                              │
  │    ┌────────────────────────────────────┐   │
  │    │ a. 检查文件是否存在                  │   │
  │    │ b. 压缩/生成缩略图（如需要）          │   │
  │    │ c. POST /api/file/upload            │   │
  │    │    multipart/form-data             │──►│
  │    │    ├── file: 原始图                 │   │
  │    │    ├── thumbnail: 缩略图(可选)      │   │
  │    │    └── record_id, table_name        │   │
  │    │                                    │   │
  │    │ d. 响应: {                          │◄──│
  │    │      url: "/uploads/2024/xx.jpg",   │   │
  │    │      thumbnail_url: "...",          │   │
  │    │      file_id: 123                   │   │
  │    │    }                                 │   │
  │    │                                    │   │
  │    │ e. 更新本地记录：                     │   │
  │    │    photo_path = url                 │   │
  │    │    thumbnail_path = thumbnail_url   │   │
  │    │    (sync_status 仍为 0)              │   │
  │    └────────────────────────────────────┘   │
  │                                            │
  │ 3. 所有照片文件上传完成后，                   │
  │    触发【批量上传流程】同步 photo 记录        │
```

### 8.3 照片下载流程

```
移动端                                            服务端
  │                                                │
  │ 1. 增量/全量下载完成后，                          │
  │    检查返回的 photo_path                         │
  │                                                │
  │ 2. 收集所有未缓存的照片URL，进入下载队列           │
  │    （本地缓存目录与 photo_path 比对）             │
  │                                                │
  │ 3. 逐张下载（最多3并发）：                        │
  │    GET {photo_path}  ────────────────────────►│
  │    保存到本地缓存目录                             │
  │    更新本地记录 photo_path 为本地路径             │
  │                                                │
  │ 4. 缩略图优先下载，原图按需下载                    │
  │    (列表页只显示缩略图，点详情再下载原图)          │
```

### 8.4 本地缓存策略

```
本地存储目录结构：
/storage/emulated/0/Android/data/com.excavation/
├── cache/
│   └── photos/
│       ├── thumbnails/
│       │   └── 20240617/
│       │       └── T0301-W001_1_thumb.jpg
│       └── original/
│           └── 20240617/
│               └── T0301-W001_1.jpg
└── files/
    └── ...
```

- **LRU 缓存淘汰**：超过500MB时删除最久未访问的照片
- **离线可用性**：已下载的照片不随同步删除
- **清理机制**：用户可手动「清理缓存」释放空间

## 9. 断线重试机制

### 9.1 触发重试的场景

- 网络连接超时（TCP timeout）
- 服务端返回 5xx 错误
- 网络切换（WiFi ↔ 移动数据）
- DNS 解析失败

### 9.2 指数退避算法

```
重试间隔 = base_delay * (2 ^ retry_count) + random_jitter

参数：
  base_delay   = 2 秒    （基础延迟）
  max_retry    = 5 次    （最大重试次数）
  max_delay    = 60 秒   （最大延迟封顶）
  jitter_range = ±500ms  （随机抖动，避免雪崩）
```

### 9.3 重试时间表

| 重试次数 | 计算式 | 实际等待（约） |
|---------|--------|--------------|
| 第1次失败 | 2 * 2^0 + jitter | 2 秒 |
| 第2次失败 | 2 * 2^1 + jitter | 4 秒 |
| 第3次失败 | 2 * 2^2 + jitter | 8 秒 |
| 第4次失败 | 2 * 2^3 + jitter | 16 秒 |
| 第5次失败 | 2 * 2^4 + jitter | 32 秒 |
| 超过5次 | - | 标记 FAILED，停止重试 |

### 9.4 重试控制流程

```
同步请求失败
    │
    ▼
判断错误类型
    │
    ├── 4xx 错误（401/403/404/409等）
    │   └── 不重试，直接标记 FAILED
    │       error_message = 具体错误信息
    │
    └── 可重试错误（5xx/网络错误/超时）
        │
        ▼
    retry_count < 5 ?
        │
        ├── 是（继续重试）
        │   │
        │   ├── sync_record.retry_count++
        │   ├── 计算等待时间 = 2^(retry_count) * 2s + jitter
        │   ├── sync_record.last_attempt_at = now
        │   ├── sync_record.sync_status = 0（待重试）
        │   │
        │   └── 进入等待队列，时间到后重试
        │
        └── 否（超过次数）
            │
            ├── sync_record.sync_status = 3 (FAILED)
            ├── sync_record.error_message = "MAX_RETRY_EXCEEDED"
            ├── 业务表记录 sync_status = 3
            └── 通知用户：同步失败，需检查网络后手动重试
```

### 9.5 手动重试

用户在同步记录列表中看到失败的记录后，可以：

1. **单条重试**：对某条 sync_status=3 的记录单独重试
2. **批量重试**：一键重试所有失败记录
3. **重置状态**：将 sync_status=3 改回 0（下次自动同步会处理）

## 10. 同步记录追踪（sync_record表）

### 10.1 表字段说明

| 字段 | 说明 |
|------|------|
| `device_id` | 移动端设备唯一标识，如 `ANDROID_{uuid}` |
| `table_name` | 业务表名，如 `artifact` |
| `record_id` | 业务记录的主键ID |
| `operation_type` | INSERT / UPDATE / DELETE |
| `sync_direction` | UP（上传） / DOWN（下载） |
| `sync_status` | 0-待同步, 1-同步中, 2-成功, 3-失败 |
| `retry_count` | 已重试次数 |
| `error_message` | 失败原因或冲突信息 |
| `last_attempt_at` | 最后尝试时间 |
| `completed_at` | 完成时间（成功/失败） |
| `batch_id` | 所属批次号 |

### 10.2 典型查询场景

```sql
-- 查询某设备所有失败的上传记录
SELECT * FROM sync_record
WHERE device_id = 'ANDROID_xxx'
  AND sync_direction = 'UP'
  AND sync_status = 3
ORDER BY created_at DESC;

-- 查询某批次的同步进度
SELECT sync_status, COUNT(*) as cnt
FROM sync_record
WHERE batch_id = 'BATCH_ANDROID_xxx_1718000000000'
GROUP BY sync_status;

-- 清理30天前的同步记录
DELETE FROM sync_record
WHERE created_at < datetime('now', '-30 days')
  AND sync_status IN (2, 3);
```

### 11.2 异常场景处理

| 场景 | 处理方式 |
|------|---------|
| 同步过程中 App 被杀死 | 下次启动时根据 `sync_record` 中状态=1（同步中）的记录恢复 |
| 上传一半网络断开 | 已到达服务端的批次已提交，未到达的在重试时重新发送 |
| 服务端重启 | 移动端收到 5xx 错误，进入指数退避重试 |
| 设备时间不准 | 统一使用服务端 `server_timestamp`，不信任客户端时间 |
| 多人同时修改同一条记录 | 以最后成功上传到服务端的为准，其他人下载时触发冲突提示 |
